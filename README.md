**GitHub Repository：** [AWS Learning Game，雲端學習大富翁](https://github.com/ceeyu/CloudMonopoly)

## 專案摘要

身為 AWS Community Builder，在2025年，針對先前在AWS Community Day時聽到AWS BuilderCards 而受到啟發，作者把各類雲端服務做成卡牌的樣子來融入遊戲與知識的方式讓我十分喜愛，而自己在活動中也獲得一份卡牌套組也令我十分開心，而今年我也思考，恰逢AWS kiro的浪潮，我可以利用自己的身分以及這樣的資源做到什麼可以貢獻的事情?左思右想下，我想到了這個遊戲的起源，利用大家常見的大富翁遊戲，作為線上版，讓大家不被實體卡牌所束縛；並且結合不同種雲端使用者的「身分」(不同類型的企業決策者)，在自己角色不同時，決定使用的方式、服務大小也會有所不同。再結合一些觀念選擇題，讓玩家可以思考遇到不同的情景時，該用怎麼樣的服務，將AWS大大小小的服務融會貫通，而這些的參考依據也包含了AWS SAA 認證考試的學習，在遊戲中自然地理解 40+ 種 AWS 服務的應用場景與最佳實踐讓大家有寓教於樂的環境。並且這個專案從設計理念、技術實作到部署，全程使用 AWS 服務，本身就是一個 AWS 架構的實踐案例。

## 為什麼做這個專案

在社群推廣 AWS 的過程中，我觀察到幾個痛點：

1. **學習曲線陡峭**：AWS的總體服務共有 200+ 種服務，初學者不知道從何開始
2. **缺乏情境感**：讀文件和看影片很難建立「什麼時候該用什麼服務」的直覺，很難去思考為什麼同樣的功能會對應不同種的服務、價格與相關規模等
3. **SAA 認證準備枯燥**：大量的服務比較和架構模式需要反覆記憶，以及容易讓初學者進入死背硬記的迴圈中
4. **缺少互動式學習工具**：社群活動中需要能讓多人一起參與的學習方式，而不是讓一個人在厚重的規則中迷失自己
5. **相關資安意識的缺乏**：很多時候，尤其像現在AI盛行，大家很容易陷入「完成一個開發功能，可以work就好」的胡同裡，很常會忽略在一些細節中容易受到資安侵害的危機，而在遊戲中加入這些突發事件也是提醒大家，我們在設計功能或實踐時，並不是只有best case的情況，很多時候要有更完整的周全考慮

AWS Learning Game 的設計理念是「在遊戲中學習」——每一次決策都是一個真實的架構選擇題，每一次回饋都包含 AWS 最佳實踐和 SAA 考點。

## 專案亮點

### 遊戲設計

- **四種公司角色**，對應不同的企業數位轉型階段：
  - 新創公司（資本少、需快速成長）
  - 傳產公司（資源多、需完成雲端轉型）
  - 雲端代理商（技術強、需擴大規模）
  - 雲端公司（雲端原生、需兼顧成長與安全）
- **52 個實際發生的真實商業情境事件**，涵蓋：
  - 擴廠與基礎設施擴展、跨國市場拓展
  - DDoS 攻擊防護、資料外洩處理、勒索軟體應對
  - 微服務架構轉型、效能優化、災難復原
  - AI/ML 產品開發、IoT 物聯網、資料分析平台
- **每個事件提供 AWS 方案 vs 地端方案的比較**，讓玩家理解雲端的價值
- **角色專屬勝利條件**，形成不同的策略路線：
  - 新創：累積資本 ≥ 3000 萬
  - 傳產：雲端採用率 ≥ 80%
  - 代理商：員工數 ≥ 150 人
  - 雲端公司：資本 ≥ 2000 萬 且 資安等級 ≥ 5

### AWS 服務涵蓋範圍

遊戲涵蓋 **40+ 種 AWS 服務**，對應 SAA 認證六大領域：

| 領域 | 涵蓋服務 |
|------|---------|

## 技術架構（全 AWS 部署）

這個專案本身就是一個 AWS 架構實踐案例：

```
使用者瀏覽器
    │
    ▼
CloudFront (CDN/HTTPS)
    ├── S3 (React 前端靜態資源)
    │
    └── App Runner (Go 後端 API)
            │
            ▼
        DynamoDB (遊戲狀態持久化)
```

### 技術選型

| 層級 | 技術 | 說明 |
|------|------|------|
| 後端 | Go 1.23 + Gin | 高效能遊戲引擎，支援並行處理 |
| 前端 | React 18 + TypeScript + Vite | 現代化 SPA，型別安全 |
| 容器化 | Docker + ECR | 多階段建置，最小化映像檔 |
| 後端託管 | AWS App Runner | 全託管容器服務，自動擴展 |
| 前端託管 | S3 + CloudFront | 全球 CDN 加速 |
| 資料庫 | DynamoDB | 按需容量，遊戲狀態持久化 |
| CI/CD | ECR + App Runner 自動部署 | 推送映像檔即自動更新 |

### 部署成本

整個專案的 AWS 月費約 **$7-18 美元**，展示了如何用最低成本部署一個完整的全端應用。

## 軟體工程實踐

### 模組化架構

後端採用 Manager 模式，將遊戲引擎拆分為獨立模組：

- **GameEngine**：遊戲核心邏輯、回合管理、勝利判定
- **BoardManager**：棋盤建立與格子管理
- **CompanyManager**：公司建立與屬性管理
- **EventSystem**：52 個事件的管理與隨機分配
- **DecisionEngine**：決策評估、成功率計算、結果產生
- **GameStorage**：DynamoDB 持久化介面

### 屬性基礎測試 (Property-Based Testing)

專案採用 PBT 作為核心測試策略，使用 `pgregory.net/rapid` 框架定義了 **22 個正確性屬性**，涵蓋：

- 位置移動正確性（骰子值 + 棋盤繞行）
- 格子著陸資源更新精確性
- 繞圈獎勵計算
- 玩家數量驗證（2-4 人）
- 回合順序公平性
- 勝利條件判定
- 遊戲狀態序列化往返一致性
- 儲存與載入一致性

PBT 自動產生大量隨機測試案例，比傳統單元測試更能發現邊界條件的 bug。

### 完整的文件體系

專案包含三份完整的技術文件：

1. **設計規格書**：架構設計、遊戲機制、API 設計、資料模型
2. **測試規格書**：22 個 PBT 屬性定義、單元測試清單、API 整合測試、需求追溯矩陣
3. **使用者手冊**：遊戲規則、公司策略、AWS 學習指南、部署指南

## 開發歷程

### 核心功能開發

- 遊戲引擎設計與實作（Go）
- 52 個真實商業情境事件設計
- 決策引擎（資格評估、成功率計算、獎懲機制）
- React 前端（棋盤、事件彈窗、公司狀態卡片）
- REST API 設計與實作

### 勝利條件系統

- 四種公司類型的專屬勝利條件
- 勝利進度即時計算與顯示
- 30 回合限制與進度判定機制
- 遊戲規則說明頁面與勝利結算畫面

### 決策結果應用修復

- 發現並修復決策結果未正確應用到公司狀態的 bug
- 確保雲端採用率、資安等級等屬性在決策後正確更新
- 這個修復對傳產公司的遊戲體驗至關重要（需要透過 AWS 決策提升雲端採用率）

### 部署與驗證

- Docker 容器化建置
- ECR + App Runner 自動部署
- S3 + CloudFront 前端部署
- 生產環境 API 驗證

## 結論

AWS Learning Game 體現了我作為 AWS Community Builder 的核心理念：**讓 AWS 學習變得有趣且容易上手**。很多時候我在slack社群中收穫了很多新穎的知識與應用，站在時代浪潮，但隨著時間增加，我覺得更重要的是，如何讓這些資訊對大眾也變得唾手可得。畢竟，科技是用來減少人類社會間的差距，讓大家得以共榮的存在。而這個專案不只是一個遊戲，也是自己在思考如何貢獻一己之力的一份心力。同樣像開源的社群提供自己的努力。而這個專案除了完整的 AWS 架構實踐案例——從後端的 Go 服務到前端的 React 應用，從 App Runner 部署到 DynamoDB 持久化，從 Property-Based Testing 到 CI/CD 流程，每一個環節都展示了 AWS 服務的實際應用。而這不但讓人們可以獲得輕鬆學習的環境，對我而言也體會到kiro的強大之處。在我尚不熟悉的語言/框架中，仍能有如此優異與順暢的體驗，我十分的雀躍與大為震撼，而我也期待在接下來的 Community Builder 任期中，將這個專案帶到更多的社群活動中，變得更臻完善，幫助更多人以輕鬆的方式踏入 AWS 雲端架構的世界。
