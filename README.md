## 專案摘要

身為 AWS Community Builder，在2025年，針對先前在AWS Community Day時聽到AWS BuilderCards 而受到啟發，作者把各類雲端服務做成卡牌的樣子來融入遊戲與知識的方式讓我十分喜愛，而自己在活動中也獲得一份卡牌套組也令我十分開心，而今年我也思考，恰逢AWS kiro的浪潮，我可以利用自己的身分以及這樣的資源做到什麼可以貢獻的事情?左思右想下，我想到了這個遊戲的起源，利用大家常見的大富翁遊戲，作為線上版，讓大家不被實體卡牌所束縛；並且結合不同種雲端使用者的「身分」(不同類型的企業決策者)，在自己角色不同時，決定使用的方式、服務大小也會有所不同。再結合一些觀念選擇題，讓玩家可以思考遇到不同的情景時，該用怎麼樣的服務，將AWS大大小小的服務融會貫通，而這些的參考依據也包含了AWS SAA 認證考試的學習，在遊戲中自然地理解 40+ 種 AWS 服務的應用場景與最佳實踐讓大家有寓教於樂的環境。並且這個專案從設計理念、技術實作到部署，全程使用 AWS 服務，本身就是一個 AWS 架構的實踐案例。

## 為什麼做這個專案

在社群推廣 AWS 的過程中，我觀察到幾個痛點：

1. **學習曲線陡峭**：AWS的總體服務共有 200+ 種服務，初學者不知道從何開始
2. **缺乏情境感**：讀文件和看影片很難建立「什麼時候該用什麼服務」的直覺，很難去思考為什麼同樣的功能會對應不同種的服務、價格與相關規模等
3. **SAA 認證準備枯燥**：大量的服務比較和架構模式需要反覆記憶，以及容易讓初學者進入死背硬記的迴圈中
4. **缺少互動式學習工具**：社群活動中需要能讓多人一起參與的學習方式，而不是讓一個人在厚重的規則中迷失自己
5. **相關資安意識的缺乏**：很多時候，尤其像現在AI盛行，大家很容易陷入「完成一個開發功能，可以work就好」的胡同裡，很常會忽略在一些細節中容易受到資安侵害的危機，而在遊戲中加入這些突發事件也是提醒大家，我們在設計功能或實踐時，並不是只有best case的情況，很多時候要有更完整的周全考慮

AWS Learning Game 的設計理念是「在遊戲中學習」——每一次決策都是一個真實的架構選擇題，每一次回饋都包含 AWS 最佳實踐和 SAA 考點。

## 技術架構（全 AWS 部署）

這個專案本身就是一個 AWS 架構實踐案例：

```
使用者瀏覽器
    │
    ▼
CloudFront (CDN/HTTPS)
    ├── S3 (React 前端靜態資源)
    │
    └── App Runner (Go 後端 API)
            │
            ▼
        DynamoDB (遊戲狀態持久化)
```

### 技術選型

| 層級 | 技術 | 說明 |
|------|------|------|
| 後端 | Go 1.23 + Gin | 高效能遊戲引擎，支援並行處理 |
| 前端 | React 18 + TypeScript + Vite | 現代化 SPA，型別安全 |
| 容器化 | Docker + ECR | 多階段建置，最小化映像檔 |
| 後端託管 | AWS App Runner | 全託管容器服務，自動擴展 |
| 前端託管 | S3 + CloudFront | 全球 CDN 加速 |
| 資料庫 | DynamoDB | 按需容量，遊戲狀態持久化 |
| CI/CD | ECR + App Runner 自動部署 | 推送映像檔即自動更新 |

### 部署成本

整個專案的 AWS 月費約 **$7-18 美元**，展示了如何用最低成本部署一個完整的全端應用。

## 軟體工程實踐

### 模組化架構

後端採用 Manager 模式，將遊戲引擎拆分為獨立模組：

- **GameEngine**：遊戲核心邏輯、回合管理、勝利判定
- **BoardManager**：棋盤建立與格子管理
- **CompanyManager**：公司建立與屬性管理
- **EventSystem**：52 個事件的管理與隨機分配
- **DecisionEngine**：決策評估、成功率計算、結果產生
- **GameStorage**：DynamoDB 持久化介面

### 屬性基礎測試 (Property-Based Testing)

專案採用 PBT 作為核心測試策略，使用 `pgregory.net/rapid` 框架定義了 **22 個正確性屬性**，涵蓋：

- 位置移動正確性（骰子值 + 棋盤繞行）
- 格子著陸資源更新精確性
- 繞圈獎勵計算
- 玩家數量驗證（2-4 人）
- 回合順序公平性
- 勝利條件判定
- 遊戲狀態序列化往返一致性
- 儲存與載入一致性

PBT 自動產生大量隨機測試案例，比傳統單元測試更能發現邊界條件的 bug。

### 完整的文件體系

專案包含三份完整的技術文件：

1. **設計規格書**：架構設計、遊戲機制、API 設計、資料模型
2. **測試規格書**：22 個 PBT 屬性定義、單元測試清單、API 整合測試、需求追溯矩陣
3. **使用者手冊**：遊戲規則、公司策略、AWS 學習指南、部署指南

## 1. 遊戲簡介

### 1.1 什麼是 AWS Learning Game？

AWS Learning Game 是一款結合大富翁遊戲機制與 AWS 雲端架構學習的多人線上桌遊。你將扮演一家公司的決策者，在棋盤上前進、遭遇各種商業情境，並做出 AWS 架構決策。透過遊戲，你可以自然地學習 AWS Solutions Architect Associate (SAA) 認證所需的知識。

### 1.2 遊戲特色

- **大富翁式棋盤遊戲**：擲骰子、前進、觸發事件
- **四種公司角色**：新創、傳產、雲端代理商、雲端公司
- **52 個真實商業情境**：擴廠、跨國合作、資安事件等
- **40+ 種 AWS 服務**：涵蓋 SAA 認證六大領域
- **即時學習回饋**：每次決策都提供學習要點與最佳實踐
- **2-4 人多人對戰**：與朋友一起學習、競爭

## 4. 遊戲規則

### 4.1 基本流程

1. 玩家輪流擲骰子（1-6 點）
![image](https://hackmd.io/_uploads/Hy6NrIC6be.png)
2. 根據點數在棋盤上前進
3. 踏上不同格子會觸發不同效果
4. 做出決策後，系統計算結果並更新公司狀態
5. 先達成勝利條件的玩家獲勝

### 4.2 棋盤格子類型

| 格子 | 圖示 | 效果 |
|------|------|------|
| 起點 | 🏁 | 遊戲起點 |
| 一般 | 📍 | 獲得基礎資源（+100 萬資本、+3 員工） |
| 機會 | 🌟 | 觸發機會事件（擴廠、合作等正面機會） |
| 命運 | 🎭 | 觸發命運事件（市場變化、人才流失等） |
| 關卡 | ⚔️ | 觸發技術挑戰（架構轉型、效能優化） |
| 獎勵 | 🎁 | 獲得額外資源獎勵 |

### 4.3 繞圈獎勵

當你的棋子繞行棋盤一圈（經過起點）時，會獲得額外獎勵：
- 💰 +200 萬資本
- 👥 +5 員工

### 4.4 回合限制

- 每位玩家最多 **30 回合**
- 若 30 回合內無人達成勝利條件，根據**勝利進度百分比**判定贏家
- 勝利進度越高的玩家獲勝

### 4.5 勝利進度計算

你可以在玩家狀態卡片上看到勝利進度條：

- **新創公司**：進度 = 當前資本 ÷ 3000 × 100%
- **傳產公司**：進度 = 當前雲端採用率 ÷ 80 × 100%
- **雲端代理商**：進度 = 當前員工數 ÷ 150 × 100%
- **雲端公司**：進度 = (資本進度 + 資安進度) ÷ 2 × 100%

遊戲包含 52 個精心設計的事件，基於真實商業情境：

**機會事件（🌟）：**
- 擴廠與基礎設施擴展
- 跨國市場拓展
- AI/ML 產品開發

### 5.2 如何做決策

當你踏上事件格子時，會彈出事件視窗：

1. **閱讀情境描述**：了解你面臨的商業情境
2. **查看選項**：通常有 AWS 雲端方案和地端方案
3. **比較需求條件**：確認你的公司是否符合條件
4. **查看預期結果**：資本變化、員工變化、成功率等
5. **選擇方案**：點選你認為最佳的方案
6. **確認決策**：點擊「確認決策」提交

### 5.3 決策結果

提交決策後，系統會顯示：

- ✅/⚠️ **成功或失敗**：基於成功率和公司屬性計算
- 📊 **實際影響**：資本、員工、資安、雲端採用率的變化
- 📚 **學習要點**：與此決策相關的 AWS 知識
- 💡 **AWS 最佳實踐**：專業建議

### 5.4 影響成功率的因素

| 因素 | 影響 |
|------|------|
| 資本充裕（超過需求 2 倍） | 成功率 +10% |
| 員工充裕（超過需求 2 倍） | 成功率 +5% |
| 資安等級超過需求 | 成功率 +5% |
| 雲端採用率 > 50%（AWS 方案） | 成功率 +10% |
| 技術債 > 50 | 成功率 -10% |

## 7. 學習指南

### 7.1 AWS 服務學習路徑

遊戲涵蓋 40+ 種 AWS 服務，以下是建議的學習重點：

**運算服務：**
- EC2：虛擬伺服器，了解執行個體類型和定價模式
- Lambda：無伺服器運算，了解事件驅動架構
- ECS/EKS：容器服務，了解微服務部署

**儲存服務：**
- S3：物件儲存，了解儲存類別和生命週期
- EBS：區塊儲存，了解磁碟區類型

**資料庫服務：**
- RDS：關聯式資料庫，了解 Multi-AZ 和讀取副本
- DynamoDB：NoSQL 資料庫，了解容量模式
- Aurora：高效能資料庫，了解自動擴展

**網路服務：**
- VPC：虛擬網路，了解子網路和安全群組
- CloudFront：CDN，了解內容分發
- Route 53：DNS 服務，了解路由策略

**安全服務：**
- IAM：身分管理，了解最小權限原則
- WAF：Web 防火牆，了解 DDoS 防護
- GuardDuty：威脅偵測，了解安全監控

### 7.2 遊戲中的學習技巧

1. **仔細閱讀事件描述**：每個事件都基於真實商業情境
2. **比較 AWS 和地端方案**：理解雲端的優勢和適用場景
3. **注意 AWS 考點標籤**：事件視窗會顯示相關的 SAA 考試主題
4. **閱讀決策回饋**：每次決策後的學習要點和最佳實踐很有價值
5. **嘗試不同公司類型**：每種公司面臨不同的挑戰，可以學到不同面向

## 8. 進階策略

### 8.1 新創公司策略

- 優先選擇高資本報酬的決策
- 善用 AWS 服務降低初期成本
- 避免高風險決策（資本不足時失敗代價大）

### 8.2 傳產公司策略

- 積極選擇 AWS 方案提升雲端採用率
- 每次 AWS 決策可增加 5-30% 的雲端採用率
- 注意技術債較高，可能降低成功率

### 8.3 雲端代理商策略

- 選擇能增加員工數的決策
- 維持資本穩定以支撐團隊擴張
- 善用高雲端採用率帶來的成功率加成

### 8.4 雲端公司策略

- 同時關注資本成長和資安提升
- 選擇能增加資安等級的決策（如安全服務相關事件）
- 利用低技術債的優勢提高決策成功率

## 9. 常見問題

### Q1: 遊戲最多可以幾個人玩？
A: 2-4 人。

### Q2: 一場遊戲大約需要多久？
A: 每位玩家最多 30 回合，2 人遊戲約 20-30 分鐘，4 人遊戲約 40-60 分鐘。

### Q3: 如果 30 回合內沒有人達成勝利條件怎麼辦？
A: 遊戲會根據勝利進度百分比判定贏家，進度最高的玩家獲勝。

### Q4: 決策失敗會怎樣？
A: 失敗時只會獲得一半的預期收益，不會造成額外損失。但如果不符合需求條件強行執行，可能會有額外懲罰。

### Q5: 可以儲存遊戲進度嗎？
A: 可以，遊戲支援儲存和載入功能（需要後端配置 DynamoDB）。

### Q6: 選擇地端方案會影響雲端採用率嗎？
A: 部分地端方案會降低雲端採用率（-5% 到 -10%），而 AWS 方案通常會增加雲端採用率（+5% 到 +30%）。

## 11. 本地開發與部署

### 11.1 環境需求

| 工具 | 版本 | 用途 |
|------|------|------|
| Go | 1.23+ | 後端開發 |
| Node.js | 18+ | 前端開發 |
| Docker | 最新版 | 容器化部署 |
| AWS CLI | 最新版 | AWS 資源管理 |

### 11.2 本地啟動

**啟動後端：**
```bash
# 安裝依賴
go mod download

# 執行測試
go test ./... -v

# 建置並啟動
go build -o server.exe ./cmd/server
./server.exe
# 後端將在 http://localhost:8080 啟動
```

**啟動前端：**
```bash
cd web
npm install
npm run dev
# 前端將在 http://localhost:3000 啟動
# API 請求會自動代理到 http://localhost:8080
```

### 11.3 部署到 AWS

使用部署腳本一鍵部署：
```powershell
.\deploy\deploy.ps1
```

此腳本會自動：
1. 建立/檢查 ECR Repository
2. 登入 ECR
3. 建置並推送 Docker image
4. 建立/檢查 S3 Bucket
5. 建置前端
6. 上傳前端到 S3

部署完成後，需要在 AWS Console 中：
- 建立 App Runner 服務（使用 ECR 映像檔）
- 建立 CloudFront Distribution（指向 S3）
- （可選）建立 DynamoDB 表格用於遊戲存檔

### 11.4 部署架構

```
使用者 → CloudFront → S3（前端靜態資源）
                  ↘
                   App Runner（Go 後端 API）→ DynamoDB（遊戲存檔）
```

**預估月費：** $7-18 美元（低流量場景）
