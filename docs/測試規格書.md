## 2. 測試策略

### 2.1 屬性基礎測試 (Property-Based Testing)

PBT 是本專案的核心測試方法。與傳統單元測試不同，PBT 不是測試特定的輸入/輸出組合，而是定義系統必須滿足的「屬性」（不變量），然後由框架自動產生大量隨機測試案例來驗證這些屬性。

**使用的 PBT 框架：** `pgregory.net/rapid`

**PBT 的優勢：**
- 自動探索邊界條件和極端情況
- 發現開發者未預見的 bug
- 失敗時自動縮小反例（shrinking）
- 失敗案例可重現（儲存於 `testdata/rapid/` 目錄）

**PBT 測試模式：**
```go
rapid.Check(t, func(t *rapid.T) {
    // 1. 產生隨機輸入
    value := rapid.IntRange(0, 100).Draw(t, "value")
    
    // 2. 執行操作
    result := SomeFunction(value)
    
    // 3. 驗證屬性（不變量）
    if result < 0 {
        t.Errorf("結果不應為負數")
    }
})
```

### 2.2 測試分層

```
┌─────────────────────────────────────┐
│     整合測試 (Integration Tests)      │  ← API 端點完整流程
├─────────────────────────────────────┤
│     屬性基礎測試 (Property Tests)     │  ← 遊戲邏輯正確性
├─────────────────────────────────────┤
│     單元測試 (Unit Tests)             │  ← 個別函數行為
└─────────────────────────────────────┘
```

## 3. 屬性基礎測試清單

本專案定義了 22 個正確性屬性，涵蓋遊戲的所有核心機制：

### Property 2: 位置移動正確性

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty2_PositionMovementCorrectness`  
**驗證需求：** Requirements 2.2

**屬性定義：**
> 對於任意骰子值 D 和當前位置 P，在大小為 S 的棋盤上，新位置必須等於 `(P + D) mod S`，確保玩家正確繞行棋盤。

**測試方法：**
- 隨機產生 1-20 個回合
- 每回合記錄移動前位置
- 驗證移動後位置 = `(舊位置 + 骰子值) % 棋盤大小`
- 驗證玩家實際位置與回合結果一致

### Property 3: 格子著陸資源更新

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty3_CellLandingResourceUpdate`  
**驗證需求：** Requirements 2.3, 2.4

**屬性定義：**
> 對於任意格子著陸事件，公司的資本必須精確增加該格子的 BaseCapital 值，員工數必須精確增加該格子的 BaseEmployees 值。

**測試方法：**
- 隨機執行 5-30 個回合
- 每回合記錄著陸前的資本與員工數
- 計算預期變化（格子基礎獎勵 + 繞圈獎勵）
- 驗證實際變化與預期完全一致

### Property 5: 繞圈完成獎勵

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty5_CircuitCompletionBonus`  
**驗證需求：** Requirements 2.6

**屬性定義：**
> 對於任意導致位置繞回起點的移動，必須額外增加繞圈獎勵（+200 萬資本、+5 員工）。

**測試方法：**
- 隨機執行 10-50 個回合（增加觸發繞圈的機率）
- 判斷是否繞圈：`(舊位置 + 骰子值) >= 棋盤大小`
- 繞圈時驗證 `CircuitCompleted` 旗標為 true
- 驗證獎勵金額正確（繞圈獎勵 + 格子獎勵）

### Property 17: 遊戲結束摘要

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty17_GameEndSummary`  
**驗證需求：** Requirements 6.3

**屬性定義：**
> 對於任意達到「finished」狀態的遊戲，必須能產生包含所有玩家決策記錄的摘要。

**測試方法：**
- 隨機產生 2-4 位玩家
- 隨機執行 5-20 個回合
- 結束遊戲後取得摘要
- 驗證摘要包含所有玩家資訊、決策記錄、贏家判定

### Property 18: 玩家數量驗證

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty18_PlayerCountValidation`  
**驗證需求：** Requirements 7.1

**屬性定義：**
> 對於任意遊戲建立或加入嘗試，遊戲引擎必須強制玩家數量在 2-4 人之間。

**測試方法：**
- 隨機產生 2-4 的最大玩家數
- 嘗試加入 1 到 maxPlayers+1 位玩家
- 驗證未超過上限時加入成功
- 驗證超過上限時返回 `ErrGameFull`

### Property 19: 回合順序公平性

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty19_TurnOrderFairness`

**屬性定義：**
> 對於任意遊戲，所有玩家的回合數差異不超過 1。

### Property 20: 贏家判定

**檔案：** `internal/game/game_test.go`  
**函數：** `TestProperty20_WinnerDetermination`

**屬性定義：**
> 對於任意結束的遊戲，必須能判定一位贏家。

### Property 21: 遊戲狀態序列化往返

**檔案：** `internal/game/persistence_test.go`  
**函數：** `TestProperty21_GameStateSerializationRoundTrip`  
**驗證需求：** Requirements 8.4, 8.5

**屬性定義：**
> 對於任意有效的遊戲狀態，序列化為 JSON 後再反序列化，必須產生與原始狀態等價的遊戲狀態。

**測試方法：**
- 隨機產生完整遊戲狀態（ID、配置、玩家、棋盤、決策歷史）
- 序列化為 JSON
- 反序列化回遊戲狀態
- 使用 `GameStateEquals` 函數比較所有欄位

### Property 22: 儲存與載入一致性

**檔案：** `internal/storage/storage_test.go`  
**函數：** `TestProperty22_SaveAndLoadConsistency`

**屬性定義：**
> 對於任意遊戲狀態，儲存後載入必須得到相同的狀態。

## 8. 生產環境驗證

### 8.1 部署後驗證步驟

1. **健康檢查**：`GET /api/v1/health` 回應 200
2. **建立遊戲**：`POST /api/v1/games` 回應 201
3. **加入遊戲**：`POST /api/v1/games/:id/join` 回應 200
4. **開始遊戲**：`POST /api/v1/games/:id/start` 回應 200
5. **執行回合**：`POST /api/v1/games/:id/turn` 回應包含 `victory_progress`
6. **提交決策**：`POST /api/v1/games/:id/decision` 回應包含 `cloud_adoption_change`
7. **驗證勝利欄位**：`GET /api/v1/games/:id` 回應包含 `max_turns_per_player: 30`

### 8.2 已知問題與修復

| 問題 | 狀態 | 修復內容 |
|------|------|---------|
| 決策結果未應用到公司狀態 | ✅ 已修復 | 在 `SubmitDecision` handler 中加入公司屬性更新邏輯 |
