## 1. 遊戲簡介

### 1.1 什麼是 AWS Learning Game？

AWS Learning Game 是一款結合大富翁遊戲機制與 AWS 雲端架構學習的多人線上桌遊。你將扮演一家公司的決策者，在棋盤上前進、遭遇各種商業情境，並做出 AWS 架構決策。透過遊戲，你可以自然地學習 AWS Solutions Architect Associate (SAA) 認證所需的知識。

### 1.2 遊戲特色

- 🎲 **大富翁式棋盤遊戲**：擲骰子、前進、觸發事件
- 🏢 **四種公司角色**：新創、傳產、雲端代理商、雲端公司
- 🌟 **52 個真實商業情境**：擴廠、跨國合作、資安事件等
- ☁️ **40+ 種 AWS 服務**：涵蓋 SAA 認證六大領域
- 📚 **即時學習回饋**：每次決策都提供學習要點與最佳實踐
- 👥 **2-4 人多人對戰**：與朋友一起學習、競爭

## 2. 快速開始

### 2.1 進入遊戲

開啟瀏覽器，前往遊戲網址。你會看到遊戲大廳頁面。

### 2.2 建立遊戲

1. 點擊「🆕 建立新遊戲」
2. 選擇玩家人數（2-4 人）
3. 點擊「建立遊戲」
4. 系統會自動產生遊戲 ID，並進入加入遊戲畫面

### 2.3 加入遊戲

1. 如果是建立者，遊戲 ID 會自動填入
2. 如果是其他玩家，輸入遊戲 ID
3. 輸入你的玩家名稱
4. 選擇公司類型（詳見第 3 節）
5. 點擊「加入遊戲」

### 2.4 開始遊戲

所有玩家加入後，任一玩家點擊「開始遊戲」即可開始。遊戲開始時會顯示規則說明頁面。

## 4. 遊戲規則

### 4.1 基本流程

1. 玩家輪流擲骰子（1-6 點）
2. 根據點數在棋盤上前進
3. 踏上不同格子會觸發不同效果
4. 做出決策後，系統計算結果並更新公司狀態
5. 先達成勝利條件的玩家獲勝

### 4.2 棋盤格子類型

| 格子 | 圖示 | 效果 |
|------|------|------|
| 起點 | 🏁 | 遊戲起點 |
| 一般 | 📍 | 獲得基礎資源（+100 萬資本、+3 員工） |
| 機會 | 🌟 | 觸發機會事件（擴廠、合作等正面機會） |
| 命運 | 🎭 | 觸發命運事件（市場變化、人才流失等） |
| 關卡 | ⚔️ | 觸發技術挑戰（架構轉型、效能優化） |
| 獎勵 | 🎁 | 獲得額外資源獎勵 |

### 4.3 繞圈獎勵

當你的棋子繞行棋盤一圈（經過起點）時，會獲得額外獎勵：
- 💰 +200 萬資本
- 👥 +5 員工

### 4.4 回合限制

- 每位玩家最多 **30 回合**
- 若 30 回合內無人達成勝利條件，根據**勝利進度百分比**判定贏家
- 勝利進度越高的玩家獲勝

### 4.5 勝利進度計算

你可以在玩家狀態卡片上看到勝利進度條：

- **新創公司**：進度 = 當前資本 ÷ 3000 × 100%
- **傳產公司**：進度 = 當前雲端採用率 ÷ 80 × 100%
- **雲端代理商**：進度 = 當前員工數 ÷ 150 × 100%
- **雲端公司**：進度 = (資本進度 + 資安進度) ÷ 2 × 100%

## 5. 事件與決策

### 5.1 事件類型

遊戲包含 52 個精心設計的事件，基於真實商業情境：

**機會事件（🌟）：**
- 擴廠與基礎設施擴展
- 跨國市場拓展
- AI/ML 產品開發
- 行動應用開發
- 電商高峰應對

**命運事件（🎭）：**
- 市場變化與競爭
- 人才流失與招募
- 自然災害與系統故障
- 法規變更

**關卡事件（⚔️）：**
- 效能優化挑戰
- 微服務架構轉型
- 系統整合
- 資料分析平台建置

**資安事件（🔒）：**
- DDoS 攻擊防護
- 資料外洩處理
- 勒索軟體應對
- API 金鑰外洩

### 5.2 如何做決策

當你踏上事件格子時，會彈出事件視窗：

1. **閱讀情境描述**：了解你面臨的商業情境
2. **查看選項**：通常有 AWS 雲端方案和地端方案
3. **比較需求條件**：確認你的公司是否符合條件
4. **查看預期結果**：資本變化、員工變化、成功率等
5. **選擇方案**：點選你認為最佳的方案
6. **確認決策**：點擊「確認決策」提交

### 5.3 決策結果

提交決策後，系統會顯示：

- ✅/⚠️ **成功或失敗**：基於成功率和公司屬性計算
- 📊 **實際影響**：資本、員工、資安、雲端採用率的變化
- 📚 **學習要點**：與此決策相關的 AWS 知識
- 💡 **AWS 最佳實踐**：專業建議

### 5.4 影響成功率的因素

| 因素 | 影響 |
|------|------|
| 資本充裕（超過需求 2 倍） | 成功率 +10% |
| 員工充裕（超過需求 2 倍） | 成功率 +5% |
| 資安等級超過需求 | 成功率 +5% |
| 雲端採用率 > 50%（AWS 方案） | 成功率 +10% |
| 技術債 > 50 | 成功率 -10% |

## 6. 遊戲介面說明

### 6.1 遊戲主畫面

遊戲主畫面分為三個區域：

- **左側：棋盤**：環形棋盤，顯示所有格子和玩家位置
- **右側：玩家列表**：所有玩家的公司狀態卡片
- **中央：資訊區**：遊戲資訊、格子類型說明

### 6.2 玩家狀態卡片

每位玩家的狀態卡片顯示：

- 🏆 **勝利進度條**：距離勝利的百分比
- 💰 **資本額**：公司目前的資本
- 👥 **員工數**：公司目前的員工人數
- 🔒 **資安等級**：基礎/標準/進階/企業/最高
- ☁️ **雲端採用率**：雲端服務的採用百分比
- 📦 **產品週期**：開發期/上市期/成長期/成熟期
- ⚠️ **技術債**：影響決策成功率
- 📍 **位置**：目前在棋盤上的位置

### 6.3 操作按鈕

- **🎲 擲骰子**：輪到你時點擊擲骰子
- **開始遊戲**：所有玩家加入後開始遊戲
- **確認決策**：在事件視窗中提交你的選擇

## 7. 學習指南

### 7.1 AWS 服務學習路徑

遊戲涵蓋 40+ 種 AWS 服務，以下是建議的學習重點：

**運算服務：**
- EC2：虛擬伺服器，了解執行個體類型和定價模式
- Lambda：無伺服器運算，了解事件驅動架構
- ECS/EKS：容器服務，了解微服務部署

**儲存服務：**
- S3：物件儲存，了解儲存類別和生命週期
- EBS：區塊儲存，了解磁碟區類型

**資料庫服務：**
- RDS：關聯式資料庫，了解 Multi-AZ 和讀取副本
- DynamoDB：NoSQL 資料庫，了解容量模式
- Aurora：高效能資料庫，了解自動擴展

**網路服務：**
- VPC：虛擬網路，了解子網路和安全群組
- CloudFront：CDN，了解內容分發
- Route 53：DNS 服務，了解路由策略

**安全服務：**
- IAM：身分管理，了解最小權限原則
- WAF：Web 防火牆，了解 DDoS 防護
- GuardDuty：威脅偵測，了解安全監控

### 7.2 遊戲中的學習技巧

1. **仔細閱讀事件描述**：每個事件都基於真實商業情境
2. **比較 AWS 和地端方案**：理解雲端的優勢和適用場景
3. **注意 AWS 考點標籤**：事件視窗會顯示相關的 SAA 考試主題
4. **閱讀決策回饋**：每次決策後的學習要點和最佳實踐很有價值
5. **嘗試不同公司類型**：每種公司面臨不同的挑戰，可以學到不同面向

## 8. 進階策略

### 8.1 新創公司策略

- 優先選擇高資本報酬的決策
- 善用 AWS 服務降低初期成本
- 避免高風險決策（資本不足時失敗代價大）

### 8.2 傳產公司策略

- 積極選擇 AWS 方案提升雲端採用率
- 每次 AWS 決策可增加 5-30% 的雲端採用率
- 注意技術債較高，可能降低成功率

### 8.3 雲端代理商策略

- 選擇能增加員工數的決策
- 維持資本穩定以支撐團隊擴張
- 善用高雲端採用率帶來的成功率加成

### 8.4 雲端公司策略

- 同時關注資本成長和資安提升
- 選擇能增加資安等級的決策（如安全服務相關事件）
- 利用低技術債的優勢提高決策成功率

## 9. 常見問題

### Q1: 遊戲最多可以幾個人玩？
A: 2-4 人。

### Q2: 一場遊戲大約需要多久？
A: 每位玩家最多 30 回合，2 人遊戲約 20-30 分鐘，4 人遊戲約 40-60 分鐘。

### Q3: 如果 30 回合內沒有人達成勝利條件怎麼辦？
A: 遊戲會根據勝利進度百分比判定贏家，進度最高的玩家獲勝。

### Q4: 決策失敗會怎樣？
A: 失敗時只會獲得一半的預期收益，不會造成額外損失。但如果不符合需求條件強行執行，可能會有額外懲罰。

### Q5: 可以儲存遊戲進度嗎？
A: 可以，遊戲支援儲存和載入功能（需要後端配置 DynamoDB）。

### Q6: 選擇地端方案會影響雲端採用率嗎？
A: 部分地端方案會降低雲端採用率（-5% 到 -10%），而 AWS 方案通常會增加雲端採用率（+5% 到 +30%）。

## 11. 本地開發與部署

### 11.1 環境需求

| 工具 | 版本 | 用途 |
|------|------|------|
| Go | 1.23+ | 後端開發 |
| Node.js | 18+ | 前端開發 |
| Docker | 最新版 | 容器化部署 |
| AWS CLI | 最新版 | AWS 資源管理 |

### 11.2 本地啟動

**啟動後端：**
```bash
# 安裝依賴
go mod download

# 執行測試
go test ./... -v

# 建置並啟動
go build -o server.exe ./cmd/server
./server.exe
# 後端將在 http://localhost:8080 啟動
```

**啟動前端：**
```bash
cd web
npm install
npm run dev
# 前端將在 http://localhost:3000 啟動
# API 請求會自動代理到 http://localhost:8080
```

### 11.3 部署到 AWS

使用部署腳本一鍵部署：
```powershell
.\deploy\deploy.ps1
```

此腳本會自動：
1. 建立/檢查 ECR Repository
2. 登入 ECR
3. 建置並推送 Docker image
4. 建立/檢查 S3 Bucket
5. 建置前端
6. 上傳前端到 S3

部署完成後，需要在 AWS Console 中：
- 建立 App Runner 服務（使用 ECR 映像檔）
- 建立 CloudFront Distribution（指向 S3）
- （可選）建立 DynamoDB 表格用於遊戲存檔

### 11.4 部署架構

```
使用者 → CloudFront → S3（前端靜態資源）
                  ↘
                   App Runner（Go 後端 API）→ DynamoDB（遊戲存檔）
```

**預估月費：** $7-18 美元（低流量場景）
